NLP models do not consume language directly. They consume a representation contract: normalized text, token ids, masks, features, vectors, labels, and output schemas.

1. Text Contracts

Representation Typical shape Meaning Common pitfalls
Raw text string or list of strings Original document, message, question, or transcript. Hidden encoding issues, HTML markup, duplicated templates, leakage in quoted replies.
Tokens [T] Words, subwords, characters, or byte-level pieces. Train/test tokenizer mismatch, inconsistent lowercasing, losing punctuation that carries signal.
Bag-of-words features [N, V] sparse matrix Document-term counts or TF-IDF weights. Fitting vocabulary on validation data, exploding vocabulary with rare tokens.
Embedding sequence [N, T, D] Dense token vectors used by neural sequence models. Masking padded tokens incorrectly, truncating useful context.
Transformer input input_ids [N, T], attention_mask [N, T] Token ids plus mask saying which positions are real tokens. Wrong special tokens, missing causal mask, prompt text leaking labels.
LLM output tokens, JSON, tool call, embedding, or ranked documents The output must match the task contract, not just sound fluent. Evaluating style instead of correctness, unparseable JSON, unsupported claims.
Rule of thumb: write the input and output contract before choosing an NLP model. A spam classifier, a semantic search index, a summarizer, and an agentic tool caller may all use language, but they optimize different contracts.

Common preprocessing pipeline

# Classic NLP
raw_text -> normalize -> tokenize -> CountVectorizer/TfidfVectorizer -> classifier

# Transformer NLP
raw_text -> tokenizer -> input_ids + attention_mask -> transformer -> logits/embeddings/text

# Retrieval-augmented generation
documents -> chunks -> embeddings -> vector index
question -> embedding -> nearest chunks -> prompt -> generated answer + citations

BPE and LLM tokenization

Modern LLM tokenizers usually operate below the word level. Byte Pair Encoding-style tokenizers start from small units, repeatedly merge frequent adjacent pieces, and produce a vocabulary that can represent common words compactly while still handling rare words, names, code, and mixed-language text.

characters or bytes -> learned merges -> subword tokens -> token ids

2. Classic NLP ML

Classical NLP is still useful because it is fast, transparent, cheap to train, and strong on many supervised text classification problems. It also teaches the data contracts that modern systems still depend on.

Count vectorization

A count vectorizer builds a vocabulary from training text and turns each document into a sparse vector. Each column is a token or n-gram. The model never sees word order except through n-gram features.

X[d, term] = count(term in document d)

Important knobs mirror scikit-learn's CountVectorizer: lowercase, token_pattern, stop_words, ngram_range, min_df, and max_features.

TF-IDF

TF-IDF downweights terms that appear in many documents and upweights terms that distinguish a document from the corpus.

tfidf(term, doc) = tf(term, doc) * log((1 + N) / (1 + df(term))) + 1

TF-IDF often pairs well with logistic regression, linear SVMs, ridge classifiers, and Naive Bayes variants.

Classic model choices

Model Works well for Strength Watch out for
Multinomial Naive Bayes Spam, sentiment, topic labels, quick baselines. Fast, robust with counts, interpretable token evidence. Conditional independence assumption and weak handling of correlated phrases.
Logistic regression High-dimensional sparse text classification. Strong calibrated linear baseline with feature weights. Needs regularization and class imbalance handling.
Linear SVM Large-margin text classification. Often excellent with TF-IDF and n-grams. Scores are not probabilities unless calibrated.
NMF / LDA topic models Exploring unlabeled corpora. Summarizes repeated word co-occurrence patterns. Topics are not guaranteed to be semantic or stable.

3. Neural Text Representations

Neural NLP replaces sparse vocabulary columns with dense vectors. The model learns that words, subwords, sentences, and documents can be placed in continuous spaces where similarity and composition become trainable.

Embeddings

An embedding table maps token ids to dense vectors. Static embeddings such as word2vec and GloVe assign one vector per word type. Contextual embeddings from transformers assign different vectors to the same token depending on surrounding tokens.

input_ids [N, T] -> embedding_table[V, D] -> vectors [N, T, D]

Sequence models before transformers

  • CNNs: detect local n-gram features with shared filters.
  • RNNs: update a hidden state token by token.
  • LSTMs/GRUs: improve long-range memory with gates.
  • Seq2seq with attention: lets a decoder look back at encoder states.
Practical baseline: for many business text classification tasks, a TF-IDF plus logistic regression baseline should be built before fine-tuning a transformer. It exposes label quality, leakage, class imbalance, and hard examples quickly.

4. Transformers and LLMs

Transformers scale because attention lets every token exchange information with other tokens in parallel. LLMs are transformer language models trained to predict text and later adapted for instruction following, tool use, retrieval, or domain tasks.

New deep dives: open Transformers as Token Mixing Machines for attention, masks, positions, blocks, and KV cache details; then open LLMs as Probabilistic Text Systems for decoding, prompting, RAG, tools, and evaluation.

Self-attention

Each token produces a query, key, and value vector. Attention compares a query to all keys, normalizes the scores, and returns a weighted mixture of values.

attention(Q, K, V) = softmax(QK^T / sqrt(d_k) + mask) V

In encoder models, tokens can usually attend bidirectionally. In decoder language models, a causal mask prevents each token from seeing future tokens.

Transformer block

  1. Token and position embeddings enter the block.
  2. Multi-head self-attention mixes context.
  3. Residual connections and layer normalization stabilize training.
  4. A feed-forward network transforms each token independently.
  5. Stacking blocks increases abstraction and context use.

Modern NLP task patterns

Pattern Model contract Typical use Evaluation
Encoder classification Text -> pooled vector -> class logits. Intent, sentiment, moderation, entity classification. Accuracy, F1, calibration, slice metrics.
Sequence labeling Text -> per-token labels. Named entities, spans, redaction, token tagging. Entity-level precision, recall, F1.
Embeddings and retrieval Text -> dense vector. Search, clustering, deduplication, RAG. Recall@k, MRR, nDCG, answer support.
Decoder generation Prompt -> next-token distribution -> generated tokens. Chat, summarization, coding, extraction, reasoning workflows. Task-specific correctness, groundedness, format validity, human review.

7. NLP Workflow

NLP quality is dominated by data definition, split design, evaluation, and clear output contracts. Model choice matters, but it cannot repair a vague label schema or an invalid benchmark.

Stage Checklist Failure to avoid
Task framing Define the exact output: label, span, ranked docs, JSON, summary, answer, or tool call. Judging fluency when the product needs correctness or parseability.
Data cleaning Handle HTML, templates, encoding, duplicates, quoted emails, PII, language mix, and boilerplate. Letting templates dominate features or leak labels.
Splitting Split by user, thread, document family, time, or source when near duplicates exist. Same conversation or article appearing in train and validation.
Baseline Build TF-IDF plus linear model before larger neural work when labels are available. Fine-tuning first and hiding data issues behind model complexity.
LLM evaluation Use test cases, rubrics, exact checks, citation checks, format validators, and failure slices. Relying only on manual spot checks or generic preference scores.

Evaluation report minimum

  • Dataset version, split definition, text count, label distribution, source distribution, and known leakage controls.
  • Tokenizer/vectorizer settings, vocabulary size, sequence length, truncation rate, and language coverage.
  • Metrics matched to task: macro F1 for imbalanced classification, entity F1 for spans, Recall@k for retrieval, exact schema validity for structured generation.
  • Error examples grouped by cause: ambiguity, missing context, rare wording, prompt injection, unsupported answer, formatting failure.

8. Debugging NLP Systems

Most NLP bugs become obvious when you inspect the text after preprocessing and the examples behind each metric. Always keep raw text, transformed features, and model outputs traceable.

Symptom Likely cause Check
Validation score is too good Duplicate text, thread leakage, fitted vectorizer on all data, template leakage. Hash normalized text, split by source entity, fit vectorizer only on training data.
Classic model misses obvious meaning Bag-of-words lost word order or phrase meaning. Add n-grams, inspect feature weights, compare transformer embeddings.
Transformer ignores part of text Truncation, bad attention mask, prompt too long, important evidence late in context. Log token counts and truncation rate; place evidence earlier or chunk.
RAG gives unsupported answers Retriever missed evidence or generator overused prior knowledge. Measure Recall@k, show retrieved chunks, require citations, reject low-support answers.
LLM output cannot be parsed Weak schema instruction or no validation loop. Use constrained output format, schema validation, retries, and examples.
