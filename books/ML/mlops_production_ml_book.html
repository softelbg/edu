MLOps and ML Productionisation

A practical book for turning models into operated systems: labels, training, evaluation, deployment, monitoring, feedback loops, large vision-language models, and agentic production workflows.

Production ML is the discipline of keeping a model useful after it leaves a notebook. The contract includes data, labels, features, model artifacts, packages, runtime behavior, monitoring, and the decisions made from predictions.

1. Production Contracts

Contract Artifact Owner question Failure mode
Data contract Schema, source, sampling window, split policy, quality rules. Can we reproduce the exact rows and know why they were included? Training data silently differs from production input.
Label contract Guidelines, annotator policy, weak-label rules, gold set, adjudication. Does the label mean the same thing across people and time? Model learns inconsistent business policy, not ground truth.
Feature contract Transform code, feature store definitions, freshness, null policy. Are offline and online features computed the same way? Training-serving skew and hidden leakage.
Model contract Model version, hyperparameters, training code, metrics, calibration. What changed and why should this model replace the old one? Unreviewed regression hidden by one aggregate metric.
Serving contract Container, endpoint, batch job, runtime, dependencies, hardware target. Can the model meet latency, throughput, cost, and reliability needs? Good offline model cannot operate inside production constraints.
Monitoring contract Telemetry, alerts, dashboards, drift tests, feedback sampling. How do we know the model is wrong before customers tell us? Model decays quietly after data, users, or policy changes.
Rule of thumb: a model is not productionised until it has a rollback path, an owner, an evaluation report, monitored inputs and outputs, and a plan for collecting new labels.

Minimal production manifest

model:
  name: fraud_risk_v17
  training_data: transactions_2026_04_train
  evaluation_data: transactions_2026_05_holdout
  metrics: precision@alert_budget, recall, calibration, latency_p95
  package: image_sha256_or_model_registry_uri
  serving: online_endpoint_or_batch_job
  monitors: schema, drift, missingness, score_distribution, delayed_labels
  release: shadow -> canary -> 50% -> 100% with rollback

2. The Closed Production Loop

The useful mental model is a loop, not a one-way pipeline. Production data creates feedback; feedback creates labels; labels create better evaluations and retraining; release decisions create new production data.

Small model loop

  1. Define the decision and collect candidate examples.
  2. Label enough examples for a baseline and a holdout set.
  3. Train a simple model with leakage-safe splits.
  4. Evaluate against operational metrics and slices.
  5. Package preprocessing and model together.
  6. Deploy behind shadow, canary, or batch review.
  7. Monitor input drift, output drift, latency, missing values, and delayed labels.
  8. Route uncertain or high-impact cases to human review.
  9. Use reviewed cases to refresh labels and retrain.

Large model loop

  1. Define prompt, tool, retrieval, and output schema contracts.
  2. Build an eval set with exact checks and human rubrics.
  3. Choose hosted, self-hosted, quantized, or fine-tuned models.
  4. Measure latency, cost, context use, refusal, safety, and grounding.
  5. Add retrieval, validators, guardrails, and tool permission boundaries.
  6. Log prompts, retrieved evidence, tool calls, outputs, and user feedback.
  7. Sample failures into review queues and update evals before changing prompts.
label -> train -> evaluate -> package -> deploy -> monitor -> review -> relabel -> retrain

3. Production ML Use Cases

MLOps differs by the decision being automated. A nightly churn model, a real-time fraud service, a medical image triage model, and a VLLM agent need different release gates and monitors.

Use case Model pattern Production risk Key monitors
Fraud alerting Tabular classifier, graph features, rules plus ML. Adversarial drift, alert fatigue, delayed chargeback labels. Precision at alert budget, recall on delayed labels, score distribution, entity drift.
Demand forecasting Time series models, regressors, hierarchical forecasts. Calendar shifts, stockouts, promotion leakage, stale exogenous features. MAPE/MAE by horizon, bias, seasonality drift, backtest freshness.
Recommendation ranking Two-stage retrieval and ranker, embeddings, bandits. Feedback loops, popularity bias, cold-start failures. CTR, conversion, diversity, novelty, coverage, online experiment guardrails.
Computer vision inspection CNN/ViT detector or segmenter. Lighting, camera angle, product mix, annotation boundary policy. Per-class IoU/recall, false reject rate, image quality, camera drift.
Document automation OCR, layout model, LLM extraction, validators. Wrong field extraction, invalid JSON, hallucinated values. Schema validity, field-level exact match, confidence, human correction rate.
Support agent assistant RAG plus tool-calling LLM agent. Unsupported answers, unsafe tool calls, prompt injection. Answer groundedness, citation coverage, tool success, escalation rate, cost per case.
Vision-language safety review VLM classifier plus policy engine and reviewer queue. Multimodal false negatives, policy drift, demographic or context bias. Policy precision/recall, review disagreement, image source drift, appeal rate.
Autonomous coding or operations agent LLM planner, tools, sandbox, tests, approval gates. Bad tool arguments, destructive actions, hidden state, flaky tests. Task success, tool error rate, diff review findings, approval overrides, incident count.

4. Production System Patterns

The right architecture depends on latency, feedback delay, cost, risk, and how frequently the world changes.

Pattern When it fits Core design Release gate
Batch scoring Daily lists, forecasts, campaign targeting, offline ranking. Scheduled feature build, batch predict, output table, downstream checks. Data freshness, row count, score distribution, backtest score.
Online endpoint Fraud, personalization, search ranking, real-time moderation. Low-latency feature fetch, model server, request logging, fallback. Latency p95/p99, error rate, canary metrics, rollback health.
Streaming inference IoT, clicks, financial events, monitoring signals. Event bus, stateful features, streaming model, windowed monitors. Lag, exactly-once/at-least-once semantics, state correctness.
Edge deployment Mobile, robotics, cameras, low-connectivity sites. Compressed model, local runtime, telemetry sync, remote update policy. Device latency, memory, battery, rollback package, sensor coverage.
Self-hosted VLLM/LLM service Privacy, cost control, high throughput, domain fine-tunes. GPU scheduler, batching, KV cache, quantization, routing, prompt logs. Tokens/sec, queue time, context length, eval score, safety tests.
Agentic workflow Multi-step tasks with tools, retrieval, review, and state. Planner, tool registry, permissions, validators, memory, audit log. Tool-call accuracy, schema validity, human approval, sandbox tests.

7. Operating Model

MLOps is partly tooling, but mostly operating discipline. Every production model needs owners, release gates, escalation paths, and feedback ownership.

Function Responsibilities Good evidence
Product owner Defines the decision, risk tolerance, fallback, and business metric. Clear launch criteria and rollback criteria.
Data owner Maintains source definitions, schema changes, quality tests, and data access. Passing data contracts, lineage, and freshness checks.
ML engineer Builds training, evaluation, packaging, inference, and monitoring workflows. Reproducible runs, versioned artifacts, deterministic release pipeline.
Reviewer or domain expert Defines labels, audits examples, adjudicates ambiguity, and reviews failures. Label guidelines, disagreement rates, sampled production reviews.
Operations owner Handles alerts, incidents, capacity, latency, cost, and on-call procedures. Dashboards, runbooks, SLOs, incident history, capacity plan.

Release checklist

  • Training data and evaluation data are versioned and reproducible.
  • The validation split matches the deployment risk: time, user, site, camera, document family, or session.
  • Metrics include slices, not only one aggregate score.
  • The package contains preprocessing and model code together.
  • Shadow or canary telemetry is compared against the current model.
  • Rollback, fallback, human review, and incident paths are written down.
  • Monitoring covers schema, missingness, drift, score distribution, latency, errors, and delayed labels.

8. Debugging Production ML

Production failures are usually contract failures: the model sees different data, different labels, different users, different context, or different operational constraints than the evaluation assumed.

Symptom Likely cause Check
Offline score is good, production score is bad Training-serving skew, leakage, wrong split, changed population. Replay production requests through training transforms and compare feature distributions.
Model decays slowly Concept drift, user behavior shift, policy change, stale labels. Monitor delayed labels by time, segment, source, and score band.
Canary looks unstable Too little traffic, uneven routing, hidden dependency, latency tail. Check sample size, request mix, p95/p99, dependency error rates.
RAG or LLM system gives unsupported answers Retriever miss, stale index, bad chunking, prompt ignores evidence. Log query, retrieved chunks, citations, answer, validator result, and user feedback.
Agent causes bad action Tool permissions too broad, weak schema, no approval gate, hidden state. Replay tool calls, validate arguments, add sandboxing and human approval for irreversible actions. 
# Production incident triage
1. Freeze the affected model version and traffic window.
2. Pull request logs, input features, model scores, outputs, and downstream decisions.
3. Compare against the training/evaluation distribution.
4. Slice failures by source, time, class, user segment, and confidence band.
5. Decide: rollback, threshold change, fallback, hotfix, relabel, or retrain.
6. Add a regression test or monitor so the same failure is caught earlier next time.
